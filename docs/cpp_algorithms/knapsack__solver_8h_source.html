14 //
15 // This library solves knapsacks:
16 // - 0-1 knapsack problems,
17 // - Multi-dimensional knapsack problems,
18 // - TODO(user) Multi-dimensional knapsack problem with n-ary conflicts
19 // between items.
20 //
21 // Given n items, each with a profit and a weight, given a knapsack of
22 // capacity c, the goal is to find a subset of items which fits inside c
23 // and maximizes the total profit.
24 // The knapsack problem can easily be extended from 1 to d dimensions.
25 // As an example, this can be useful to constrain the maximum number of
26 // items inside the knapsack.
27 // Without loss of generality, profits and weights are assumed to be positive.
28 //
29 // From a mathematical point of view, the multi-dimensional knapsack problem
30 // can be modeled by d linear constraints:
31 // ForEach(j:1..d)(Sum(i:1..n)(weight_ij * item_i) <= c_j
32 // where item_i is a 0-1 integer variable.
33 // Then the goal is to maximize: Sum(i:1..n)(profit_i * item_i).
34 //
35 // There are several ways to solve knapsack problems. One of the most
36 // efficient ways is based on dynamic programming (mainly when weights, profits
37 // and dimensions are small, the algorithm runs in pseudo polynomial time).
38 // Unfortunately when adding conflict constraints the problem becomes strongly
39 // NP-hard, i.e. there is no pseudo-polynomial algorithm to solve it.
40 // That's the reason why the most of the following code is based on branch and
41 // bound search.
42 //
43 // For instance to solve a 2-dimension knapsack problem with 9 items,
44 // one just has to feed a profit vector with the 9 profits, a vector of 2
45 // vectors for weights, and a vector of capacities.
46 // E.g.:
47 // vector: profits = [1, 2, 3, 4, 5, 6, 7, 8, 9]
48 // vector of vector: weights = [ [1, 2, 3, 4, 5, 6, 7, 8, 9],
49 // [1, 1, 1, 1, 1, 1, 1, 1, 1]]
50 // vector: capacities = [34, 4]
51 // And then:
52 // KnapsackSolver solver(KnapsackSolver::KNAPSACK_MULTIDIMENSION_SOLVER,
53 // "Multi-dimensional solver");
54 // solver.Init(profits, weights, capacities);
55 // int64 profit = solver.Solve();

75 // ----- KnapsackSolver -----
76 // KnapsackSolver is a factory for knapsack solvers. Several solvers are
77 // implemented, some can deal with a limited number of items, some can deal with
78 // several dimensions...
79 // Currently 4 algorithms are implemented:
80 // - KNAPSACK_BRUTE_FORCE_SOLVER: Limited to 30 items and one dimension, this
81 // solver uses a brute force algorithm, ie. explores all possible states.
82 // Experiments show competitive performance for instances with less than
83 // 15 items.
84 // - KNAPSACK_64ITEMS_SOLVER: Limited to 64 items and one dimension, this
85 // solver uses a branch & bound algorithm. This solver is about 4 times
86 // faster than KNAPSACK_MULTIDIMENSION_SOLVER.
87 // - KNAPSACK_DYNAMIC_PROGRAMMING_SOLVER: Limited to one dimension, this solver
88 // is based on a dynamic programming algorithm. The time and space
89 // complexity is O(capacity * number_of_items).
90 // - KNAPSACK_MULTIDIMENSION_BRANCH_AND_BOUND_SOLVER: This solver can deal
91 // with both large number of items and several dimensions. This solver is
92 // based on branch and bound.
93 // - KNAPSACK_MULTIDIMENSION_CBC_MIP_SOLVER: This solver can deal with both
94 // large number of items and several dimensions. This solver is based on
95 // Integer Programming solver CBC.
96 // - KNAPSACK_MULTIDIMENSION_SCIP_MIP_SOLVER: This solver can deal with both
97 // large number of items and several dimensions. This solver is based on
98 // Integer Programming solver SCIP.
99 //
100 // KnapsackSolver also implements a problem reduction algorithm based on lower
101 // and upper bounds (see Ingargolia and Korsh: A reduction algorithm for
102 // zero-one single knapsack problems. Management Science, 1973). This reduction
103 // method is preferred to better algorithms (see, for instance, Martello
104 // and Toth: A new algorithm for the 0-1 knapsack problem. Management Science,
105 // 1988), because it remains valid with more complex problems, e.g.,
106 // multi-dimensional, conflicts...
107 //
108 // The main idea is to compute lower and upper bounds for each item in or out
109 // of the knapsack; if the best lower bound is strictly greater than the upper
110 // bound when an item is in, then this item is surely not in the optimal
111 // solution.
112 class BaseKnapsackSolver;

186 // The following code defines needed classes for the KnapsackGenericSolver
187 // class which is the entry point to extend knapsack with new constraints such
188 // as conflicts between items.
189 //
190 // Constraints are enforced using KnapsackPropagator objects, in the current
191 // code there is one propagator per dimension (KnapsackCapacityPropagator).
192 // One of those propagators, named master propagator, is used to guide the
193 // search, i.e. decides which item should be assigned next.
194 // Roughly speaking the search algorithm is:
195 // - While not optimal
196 // - Select next search node to expand
197 // - Select next item_i to assign (using master propagator)
198 // - Generate a new search node where item_i is in the knapsack
199 // - Check validity of this new partial solution (using propagators)
200 // - If valid, add this new search node to the search
201 // - Generate a new search node where item_i is not in the knapsack
202 // - Check validity of this new partial solution (using propagators)
203 // - If valid, add this new search node to the search
204 //
205 // TODO(user): Add a new propagator class for conflict constraint.
206 // TODO(user): Add a new propagator class used as a guide when the problem has
207 // several dimensions.

219 // ----- KnapsackItem -----
220 // KnapsackItem is a small struct to pair an item weight with its
221 // corresponding profit.
222 // The aim of the knapsack problem is to pack as many valuable items as
223 // possible. A straight forward heuristic is to take those with the greatest
224 // profit-per-unit-weight. This ratio is called efficiency in this
225 // implementation. So items will be grouped in vectors, and sorted by
226 // decreasing efficiency.
227 // Note that profits are duplicated for each dimension. This is done to
228 // simplify the code, especially the GetEfficiency method and vector sorting.
229 // As there usually are only few dimensions, the overhead should not be an
230 // issue.
231 struct KnapsackItem {
232  KnapsackItem(int _id, int64 _weight, int64 _profit)
233  : id(_id), weight(_weight), profit(_profit) {}
234  double GetEfficiency(int64 profit_max) const {
235  return (weight > 0)
236  ? static_cast<double>(profit) / static_cast<double>(weight)
237  : static_cast<double>(profit_max);
238  }
239 
240  // The 'id' field is used to retrieve the initial item in order to
241  // communicate with other propagators and state.
242  const int id;
243  const int64 weight;
244  const int64 profit;
245 };

294 // ----- KnapsackSearchPath -----
295 // KnapsackSearchPath is a small class used to represent the path between a
296 // node to another node in the search tree.
297 // As the solution state is not stored for each search node, the state should
298 // be rebuilt at each node. One simple solution is to apply all decisions
299 // between the node 'to' and the root. This can be computed in
300 // O(number_of_items).
301 //
302 // However, it is possible to achieve better average complexity. Two
303 // consecutively explored nodes are usually close enough (i.e., much less than
304 // number_of_items) to benefit from an incremental update from the node
305 // 'from' to the node 'to'.
306 //
307 // The 'via' field is the common parent of 'from' field and 'to' field.
308 // So the state can be built by reverting all decisions from 'from' to 'via'
309 // and then applying all decisions from 'via' to 'to'.
310 class KnapsackSearchPath {
311  public:
312  KnapsackSearchPath(const KnapsackSearchNode& from,
313  const KnapsackSearchNode& to);
314  void Init();
315  const KnapsackSearchNode& from() const { return from_; }
316  const KnapsackSearchNode& via() const { return *via_; }
317  const KnapsackSearchNode& to() const { return to_; }
318  const KnapsackSearchNode* MoveUpToDepth(const KnapsackSearchNode& node,
319  int depth) const;
320 
321  private:
322  const KnapsackSearchNode& from_;
323  const KnapsackSearchNode* via_; // Computed in 'Init'.
324  const KnapsackSearchNode& to_;
325 
326  DISALLOW_COPY_AND_ASSIGN(KnapsackSearchPath);
327 };

431 // ----- KnapsackCapacityPropagator -----
432 // KnapsackCapacityPropagator is a KnapsackPropagator used to enforce
433 // a capacity constraint.
434 // As a KnapsackPropagator is supposed to compute profit lower and upper
435 // bounds, and get the next item to select, it can be seen as a 0-1 Knapsack
436 // solver. The most efficient way to compute the upper bound is to iterate on
437 // items in profit-per-unit-weight decreasing order. The break item is
438 // commonly defined as the first item for which there is not enough remaining
439 // capacity. Selecting this break item as the next-item-to-assign usually
440 // gives the best results (see Greenberg & Hegerich).
441 //
442 // This is exactly what is implemented in this class.
443 //
444 // When there is only one propagator, it is possible to compute a better
445 // profit lower bound almost for free. During the scan to find the
446 // break element all unbound items are added just as if they were part of
447 // the current solution. This is used in both ComputeProfitBounds and
448 // CopyCurrentSolutionPropagator.
449 // For incrementality reasons, the ith item should be accessible in O(1). That's
450 // the reason why the item vector has to be duplicated 'sorted_items_'.
451 class KnapsackCapacityPropagator : public KnapsackPropagator {
452  public:
453  KnapsackCapacityPropagator(const KnapsackState& state, int64 capacity);
454  ~KnapsackCapacityPropagator() override;
455  void ComputeProfitBounds() override;
456  int GetNextItemId() const override { return break_item_id_; }
457 
458  protected:
459  // Initializes KnapsackCapacityPropagator (e.g., sort items in decreasing
460  // order).
461  void InitPropagator() override;
462  // Updates internal data structure incrementally (i.e., 'consumed_capacity_')
463  // to avoid a O(number_of_items) scan.
464  bool UpdatePropagator(bool revert,
465  const KnapsackAssignment& assignment) override;
466  void CopyCurrentStateToSolutionPropagator(
467  std::vector<bool>* solution) const override;
468 
469  private:
470  // An obvious additional profit upper bound corresponds to the linear
471  // relaxation: remaining_capacity * efficiency of the break item.
472  // It is possible to do better in O(1), using Martello-Toth bound U2.
473  // The main idea is to enforce integrality constraint on the break item,
474  // ie. either the break item is part of the solution, either it is not.
475  // So basically the linear relaxation is done on the item before the break
476  // item, or the one after the break item.
477  // This is what GetAdditionalProfit method implements.
478  int64 GetAdditionalProfit(int64 remaining_capacity, int break_item_id) const;
479 
480  const int64 capacity_;
481  int64 consumed_capacity_;
482  int break_item_id_;
483  std::vector<KnapsackItemPtr> sorted_items_;
484  int64 profit_max_;
485 
486  DISALLOW_COPY_AND_ASSIGN(KnapsackCapacityPropagator);
487 };
